## 2. Application Architecture

This application uses **Entity Framework Core exclusively** for all database access. There are **no raw ADO.NET SQL statements** in the codebase. All SQL is implicitly generated by EF Core LINQ operations.

The application was found in a **pre-existing PostgreSQL migration state**:
- Package references already updated to `Npgsql.EntityFrameworkCore.PostgreSQL` v8.0.0
- Connection strings already in PostgreSQL format (`Host=`, `Port=`, `Database=`)
- EF Core configuration already mapping to PostgreSQL schema (`dps_dbo.documents`)
- All column names already lowercase in EF Core mappings
- `UseNpgsql()` already in use in `Program.cs`
- `Npgsql.EnableLegacyTimestampBehavior` already set

The SQL statements were reconstructed from EF Core LINQ operations as the equivalent MS SQL Server statements the original pre-migration application would have generated.

## 7. SQL Equivalency Tool Analysis

All 7 statement pairs were validated through the SQL Equivalency tool (`sql-equivalency___validate_sql_equivalence`). All 7 returned `ERROR` status with the error message `'uniqueID'`.

Investigation revealed this is a systemic tool issue:
- The error persisted even with simplified table definitions without UUID/UNIQUEIDENTIFIER types
- The error persisted with INT-based DDLs for the Id column
- The error persisted even with completely unrelated test queries (e.g., `SELECT name FROM users WHERE id = 1`)
- Multiple retry approaches were attempted with different DDL formats

**No agent judgment was used for any equivalency determination.** All statuses are exactly as returned by the tool.
